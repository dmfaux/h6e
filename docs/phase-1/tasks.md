# Phase 1 task breakdown

Phase 1 is the first vertical slice: the logistics core a candidate and a recruiter both feel. This document slices the Phase 1 spec (`docs/phase-1/phase-1.md`) into tasks small enough to run one at a time through `/goal`, each ending in a state an agent can prove from transcript output (a test exit code, an eval-gate result, a printed audit record, an HTTP response, a git status). Each task names what it delivers and what it defers, so an unattended agent builds the seam and stops rather than pulling later-phase work forward. Delivery tasks are ordered by dependency; a realignment task follows every five delivery tasks and closes the phase.

The breakdown honours the accepted ADR register. The four Phase 1 ADRs (0009 Connect broker, 0010 verified webhook ingress, 0011 channel-bound identity, 0012 the logistics write boundary) are now Accepted, so tasks build against them directly. One decision the slicing surfaces is recorded as a new ADR in Proposed status, ADR 0026 (the WhatsApp transport, since eve 0.12 ships no native Meta adapter); Task 1.3 references it and must not wire the live Meta leg until it is Accepted.

## The external-systems reality, and how the tasks stay provable

Phase 0 left the Connect round-trip blocked: no Microsoft 365 Connect connector is provisioned and no ATS is confirmed (`docs/phase-0/task-0.6-blocker.md`, `docs/phase-0/realignment-3.md`). Phase 1 cannot assume those systems are reachable. So every task is sliced so its logic is proven by deterministic checks that always land in the transcript (`pnpm test`, `pnpm eval`, printed audit and trace records), and the live external leg (a real WhatsApp exchange, a real calendar write, a real ATS or Azure document write, a real Teams delivery) either completes against a provisioned system or records a clear blocker and stops, never faked. This is the exact pattern Task 0.6 set: prove the machinery, fail closed honestly on the live leg, and do not mark the live leg done until a real exchange is surfaced. A task with a blocked live leg can still reach `done` on its deterministic scope; its live-leg status is recorded explicitly and carried into the final realignment.

### Task 1.1: Reusable Connect scoped-credential broker
- Type: delivery
- Status: done (deterministic scope; live read blocked, no M365 connector provisioned, blocker recorded)
- Vertical slice: a caller (a workflow tool or a scheduled job) requests a named capability for one operation and receives a short-lived, least-privilege credential scoped to that operation only, with the issuance and use audit-logged; a request for a wider or write scope than the capability allows is refused; against a provisioned system the broker performs one real read, and where no system is reachable it records the blocker exactly as the Phase 0 proof path does.
- Delivers: a single internal broker over Connect that maps named capabilities (for example read calendar free/busy, write a calendar event, attach a document, read a candidate logistics field) to short-lived scoped credentials, replacing the Phase 0 one-off (F1); audit logging of every issuance and use with who, what, when, scope, target (F2); least-privilege scope enforcement so one capability cannot exceed its scope, and no decision-or-status scope is ever requested (F3, K1, ADR 0012); graceful degradation on a transient Connect failure (F4); the broker as the only path from the agent to external systems (ADR 0009).
- Defers: the live multi-system reads and writes that use the broker (Tasks 1.5 documents, 1.7 calendar); any model call (Phase 2); warehouse access (Phase 4, ADR 0022).
- Depends on: none (extends the Phase 0 `src/connect/proof.ts` pattern).
- Honours ADRs: 0009 (reusable broker), 0006 (minimal, non-PII reads; system of record stays put), 0012 and 0004 (no decision-or-status scope), 0005 (Connect treated as public beta).
- Evidence: `pnpm test src/connect` exiting 0 with cases for capability-to-scope mapping, least-privilege refusal of an over-broad or write scope on a read capability, refusal of any decision-or-status scope, and audit-record shape on issuance and use; a run against a provisioned system showing one real brokered read plus its audit records, or the recorded blocker if none is reachable; `git status --porcelain` scoped to the broker code.
- /goal condition:
  > Task 1.1 (reusable Connect scoped-credential broker) is complete when all ATS and calendar access in the codebase routes through a single internal broker over Connect that issues short-lived, least-privilege credentials per named capability and audit-logs every issuance and use with who, what, when, scope, and target (requirements F1, F2, F3, F4; ADR 0009). Prove it by surfacing in the transcript: (a) `pnpm test src/connect` exiting 0 with cases for capability-to-scope mapping, refusal of a write or over-broad scope requested against a read-only capability, refusal of any decision-or-status scope (ADR 0012, ADR 0004), the audit-record shape on both issuance and use, and graceful handling of a transient Connect failure; and (b) either one real brokered read against a provisioned system with its two audit records printed, or, if no system is reachable, the broker's blocking note listing the missing inputs and exiting non-zero, the same fail-closed pattern as `src/connect/proof.ts` (do not fake a response). Also surface `git status --porcelain` showing changes scoped to the broker code and its tests. Build only the broker. Do not build the document or calendar flows that use it (Tasks 1.5, 1.7), do not add a write scope beyond what a logistics capability needs, do not request any decision-or-status scope, and do not add a model call. Use pnpm, never npm, and current production package versions. Set Task 1.1's Status line in docs/phase-1/tasks.md to done only after the broker tests pass and either a real read or the recorded blocker is surfaced, and surface the updated status line. Or stop after 25 turns.
- Done when: every external-system path goes through the broker; the broker issues short-lived scoped credentials per capability, audit-logs issuance and use, refuses to exceed a capability's scope, and never requests a decision-or-status scope; a transient failure degrades gracefully; a real brokered read is shown or the blocker is recorded. The Phase 0 one-off is replaced and no long-lived static credential sits in application code.

## Assumptions

Marked as assumptions because they are defaults that can be corrected without reworking the breakdown.

- The Connect round-trip stays blocked from Phase 0 (no M365 connector, no confirmed ATS) until a system is provisioned. Every task with a live external leg is sliced to prove its logic deterministically and to record the blocker rather than fake the live leg, per the Task 0.6 pattern carried into `docs/phase-0/realignment-3.md`. A live leg's blocker does not block a task's deterministic scope from reaching done, but it is recorded and carried into Task 1.12.
- Durability is eve's native session plus `defineState` (ADR 0002), so the per-candidate durable workflow is an eve session keyed on the verified WhatsApp number, not hand-rolled Workflow SDK code. If an implementation introduces a separate durable store or bespoke workflow engine, that is a new decision needing an ADR (noted in Task 1.2).
- The WhatsApp transport is a custom eve channel over Meta's WhatsApp Business Cloud API (ADR 0026, Proposed). Task 1.3 builds the transport-agnostic verified-ingress and acknowledgement logic now and gates the live Meta leg on ADR 0026 being Accepted and a number being provisioned.
- Phase 1 makes no model calls. The candidate conversation is deterministic (menus, quick replies, curated FAQ) with easy human handover, per the spec's resolved open question. Any model call would need the Phase 2 gateway, redaction, and allowlist and an ADR.
- The pilot defaults are the spec's: recruiter digest weekdays 07:30 SAST; nudges first at 24h then every 48h up to 3 then escalate; quiet hours 20:00 to 07:00 SAST and Sundays and SA public holidays; stall thresholds candidate 72h, stakeholder 48h. All configurable (G4).
- If the ATS exposes no usable API, the pilot falls back to Microsoft 365 plus a designated Azure or SharePoint document store as the system of record. That store selection is a new architectural decision needing an ADR before it is built (noted in Task 1.5), not a silent default.
- The eval gate is the Phase 0 `scripts/eval-gate.mjs` plus `*.eval.ts` cases under `evals/`; the Phase 1 logistics checks are deterministic assertions, not model-judged, consistent with J3 ("deterministic logistics evals") and the no-model boundary.
- Phase 1 runs in a non-production pilot environment because the dev provider refuses production (ADR 0003). The Entra-fronted production environment is lit at Phase 5.
